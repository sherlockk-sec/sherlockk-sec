üè¢ Subnets

  • Range of IP addresses in VPC
  • Public Subnet: Internet access via IGW
  • Private Subnet: No direct internet access
  • Each subnet in one AZ

üó∫Ô∏è Route Tables

  • Define where network traffic goes
  • Every subnet must be associated
  • Local routes + custom routes
  • Main route table as default

üåê Internet Gateway (IGW)

  • Allows internet communication
  • Horizontally scaled, highly available
  • One IGW per VPC
  • Required for public subnets

üîÑ NAT Gateway

  • Outbound internet for private subnets
  • Managed by AWS
  • Placed in public subnet
  • High availability within AZ

üîÑ VPC Traffic Flow

1. Public Subnet ‚Üí Internet Gateway ‚Üí Internet
2. Private Subnet ‚Üí NAT Gateway ‚Üí Internet Gateway ‚Üí Internet
3. VPC to VPC ‚Üí VPC Peering or Transit Gateway
4. On-premises ‚Üí VPN Gateway or Direct Connect

üõ°Ô∏è Security Groups vs Network ACLs

Feature Security Groups Network ACLs
Level Instance level (ENI) Subnet level
Rules Allow rules only Allow & Deny rules
State Stateful (return traffic automatic) Stateless (must specify both directions)
Rule Processing All rules evaluated Rules processed in order
Application Applied when launching instance Automatically applies to all instances
Security Groups = Instance firewall (Allow only, Stateful). NACLs = Subnet firewall (Allow/Deny, Stateless). Both provide defense in depth!

üîó VPC Advanced Features

ü§ù VPC Peering

  • Connect two VPCs privately
  • Same or different AWS accounts
  • Same or different regions
  • Non-transitive - must peer each VPC directly
  • No overlapping CIDR blocks

‚ö° AWS Direct Connect

  • Dedicated network connection
  • On-premises to AWS
  • Consistent bandwidth & latency
  • Reduce network costs
  • Private connectivity

üîó VPC Endpoints

  • Private connection to AWS services
  • No internet gateway needed
  • Gateway Endpoints: S3, DynamoDB
  • Interface Endpoints: Most other services
  • Enhanced security

üåê Transit Gateway

  • Central hub for multiple VPCs
  • Simplify network architecture
  • Connect thousands of VPCs
  • On-premises connectivity
  • Route table control
VPC Peering is 1:1 connection. Transit Gateway is hub-and-spoke model for multiple connections. Direct Connect is for dedicated on-premises connectivity!

üéØ KEY EXAM POINTS TO REMEMBER:

  • Route 53: DNS service with multiple routing policies (Simple, Weighted, Failover, etc.)
  • API Gateway: Managed API service, supports REST/WebSocket/HTTP APIs
  • VPC: Private network in AWS, contains subnets, route tables, gateways
  • Subnets: Public (with IGW) vs Private (with NAT Gateway for outbound)
  • Security Groups: Instance-level firewall, stateful, allow rules only
  • NACLs: Subnet-level firewall, stateless, allow/deny rules
  • IAM: Global service, Users/Groups/Roles/Policies, least privilege principle
  • IAM Roles: For applications/services, temporary credentials, cross-account access
  • ACM: Free SSL/TLS certificates, automatic renewal, integrates with AWS services
  • KMS: Encryption key management, AWS-managed vs Customer-managed keys
  • VPC Peering: Connect VPCs privately, non-transitive, no overlapping CIDRs
  • Direct Connect: Dedicated connection from on-premises to AWS
  • VPC Endpoints: Private connectivity to AWS services without internet
  • NAT Gateway vs NAT Instance: Managed vs self-managed outbound internet

üî¢ Important Numbers

  • Route 53: Named after DNS port 53
  • CIDR /16: 65,536 IP addresses
  • CIDR /24: 256 IP addresses
  • VPC: Max 5 per region (soft limit)
  • Security Groups: Max 2,500 per VPC

‚ùå Common Mistakes to Avoid

  • Confusing Security Groups with NACLs
  • Thinking VPC Peering is transitive
  • Forgetting NAT Gateway for private subnet outbound
  • Not understanding IAM Roles vs Users
  • Mixing up Route 53 routing policies

‚úÖ Must Remember

  • IAM is global, not regional
  • Route 53 can route to AWS and non-AWS resources
  • Security Groups are stateful firewalls
  • VPC Endpoints keep traffic within AWS network
  • KMS keys never leave AWS unencrypted
