üíæ Amazon EBS (Elastic Block Store)

üì¶ Amazon EBS

Persistent Block Storage for EC2

  • Network-attached storage
  • Independent of EC2 lifecycle
  • Can attach/detach from instances
  • Automatic replication within AZ

‚ö° Instance Store

Temporary Block Storage

  • Physically attached to host
  • High performance
  • Data lost when instance stops
  • No additional cost

üì∏ EBS Snapshots

  • Point-in-time backup of EBS volumes
  • Stored in Amazon S3
  • Incremental backups - only changed blocks
  • Can create volumes from snapshots
  • Cross-region copy support
  • Enable disaster recovery

üí∞ EBS Pricing

  • Pay for provisioned storage per GB/month
  • Additional cost for IOPS (io1/io2)
  • Snapshot storage charged separately
  • Data transfer for cross-AZ
EBS = Persistent storage that survives instance termination. Instance Store = Temporary storage that's lost when instance stops. Choose based on data persistence needs!

üîÑ S3 Lifecycle Management

  • Transition actions: Move objects between storage classes
  • Expiration actions: Delete objects after specified time
  • Rules-based: Apply to prefixes, tags, or entire bucket
  • Cost optimization: Automatically reduce storage costs
  • Example: Standard ‚Üí IA (30 days) ‚Üí Glacier (60 days)

üîß How S3 Works

  • Buckets: Containers for objects (must be globally unique name)
  • Objects: Files stored in buckets (up to 5TB per object)
  • Keys: Unique identifier for objects within bucket
  • Regions: Buckets created in specific AWS region
  • Access: Via AWS Console, CLI, SDKs, or REST API
S3 is like a massive online storage locker. Objects go in buckets, and lifecycle policies automatically move old objects to cheaper storage classes over time!

üê≥ AWS Fargate (Serverless Containers)

üì¶ What is AWS Fargate?

  • Serverless compute engine for containers
  • No EC2 instances to manage
  • Pay-as-you-go for resources consumed
  • Works with Amazon ECS and EKS
  • Focus on applications, not infrastructure

üìã Container Basics for Fargate

  • Container: Package with code and all dependencies
  • Container Image: Blueprint created from Dockerfile
  • Task Definition: JSON file defining containers to run
  • Task: Running instance of task definition
  • Cluster: Logical grouping of resources

‚ö° AWS Lambda

  • Function-based
  • 15-minute maximum runtime
  • Event-driven execution
  • Stateless functions
  • Pay per invocation
  • Best for: Short tasks, APIs

üê≥ AWS Fargate

  • Container-based
  • Long-running applications
  • Always-on or scheduled
  • Stateful applications
  • Pay for running time
  • Best for: Web apps, microservices

üîß Fargate Key Features

  • No Infrastructure Management: AWS handles servers
  • Right-sized Resources: CPU and memory as needed
  • Secure Isolation: Each task runs in isolation
  • Integrated Security: IAM roles, VPC networking
  • Pay Per Use: Only for running containers

üéØ Fargate Use Cases

Fargate = Serverless containers for long-running applications. Lambda = Serverless functions for short-term tasks. Both eliminate server management!

üóÑÔ∏è Storage Services Complete Comparison

ü™£ Amazon S3

Object Storage

  • Access: REST API, Web Console
  • Durability: 99.999999999% (11 9s)
  • Use cases: Backup, static websites, data lake
  • Max object size: 5TB
  • Storage classes: Standard, IA, Glacier

üíæ Amazon EBS

Block Storage

  • Access: OS file system
  • Durability: 99.999% to 99.9999%
  • Use cases: OS drives, databases
  • Max volume size: 64TB (gp2), 16TB (io1)
  • Volume types: gp3, gp2, io1, io2, st1, sc1

üìÅ Amazon EFS

File Storage

  • Access: NFS protocol
  • Durability: 99.999999999% (11 9s)
  • Use cases: Shared file systems
  • Scalability: Petabyte scale
  • Concurrent access: Multiple EC2 instances

‚ö° Instance Store

Temporary Block Storage

  • Access: OS file system
  • Durability: Instance lifetime only
  • Use cases: Caching, temporary processing
  • Performance: Very high IOPS
  • Cost: Included with instance
Choose storage based on access pattern: S3 for internet access, EBS for EC2 boot/data drives, EFS for shared file access, Instance Store for temporary high-performance storage!

  • Elastic Beanstalk: PaaS, upload code and AWS handles infrastructure, 3 deployment models
  • EBS: Persistent block storage, survives instance termination, snapshots for backup
  • EBS vs Instance Store: EBS persists, Instance Store temporary (lost when stopped)
  • S3: Object storage, 11 9s durability, storage classes for cost optimization
  • S3 Storage Classes: Standard ‚Üí Standard-IA ‚Üí Glacier (decreasing cost, increasing retrieval time)
  • S3 Lifecycle: Automatically transition objects between storage classes to save costs
  • Lambda: Serverless functions, event-driven, 15-min max, pay per invocation
  • Fargate: Serverless containers, long-running apps, works with ECS/EKS
  • Lambda vs Fargate: Functions vs Containers, Short-term vs Long-running
  • Storage Types: Block (EBS), Object (S3), File (EFS), Temporary (Instance Store)

üìù Common Exam Scenarios for Module 2

üí∞ Cost Optimization

Question: How to reduce costs?

  • EC2: Use Reserved Instances for steady workloads
  • S3: Use lifecycle policies to move to cheaper storage
  • EBS: Right-size volumes, delete unused snapshots
  • Lambda: Optimize memory allocation and execution time

üìà Scaling Applications

Question: Handle variable traffic?

  • EC2: Auto Scaling Groups with load balancers
  • Beanstalk: Built-in auto scaling and load balancing
  • Lambda: Automatic scaling based on events
  • Fargate: Service auto scaling for containers

üîí Data Durability

Question: Protect against data loss?

  • EBS: Regular snapshots, cross-region copying
  • S3: Cross-region replication, versioning
  • Instance Store: Not suitable for critical data
  • Multi-AZ: Deploy across availability zones

‚ö° Performance Requirements

Question: High performance needs?

  • EC2: Choose compute-optimized instances (C5)
  • EBS: Provisioned IOPS SSD (io1/io2)
  • S3: Transfer Acceleration for global uploads
  • Lambda: Increase memory for more CPU power
Remember: The exam tests WHEN to use WHICH service for specific business requirements. Focus on understanding the trade-offs between cost, performance, and durability!
